# 旅行積立・出納帳 LINE ミニアプリ「Tabi-Pay」詳細設計・仕様書

本書は、複数回の旅行に対応し、3人グループ内での柔軟な割り勘、高精度な残高繰越、自動精算計算、およびLINEグループチャット連携を統合した、実用的な「Tabi-Pay」アプリの要件定義およびシステム詳細設計書です。

---

## 1. 企画・全体アーキテクチャ (Project Proposal)

### 1.1 背景と解決したい課題

親しい友人3人での旅行において、共通プール金（積立）と支出を管理する際、従来のツールでは以下の課題に対応できませんでした。

- **メンバー限定の支出対応**  
  「C君はお酒を飲まないが、お酒代が共通カードから一律で引き落とされる」といった不公平感。

- **複雑な現金立て替え精算**  
  旅行中に現金のみの店舗で誰かが立て替えた際、最後に「誰が誰にいくら払うべきか」の計算が極めて煩雑になる。

- **使い切り問題**  
  旅行が終わるたびに設定がリセットされ、余った小銭（繰越金）を次の旅行にスムーズに引き継げない。

- **グループ登録のハードル**  
  アプリ起動後に個別にユーザー登録や招待コードを入力するのが面倒。

- **「楽しさ」の不可視化**  
  単なる「義務的な精算」に終始してしまい、旅行全体でどれだけ贅沢をして楽しんだか、誰が何に一番お金を使ったかという「ワクワクする振り返り」ができない。

- **支出の不透明さ**  
  「これ何のお金だっけ？」と後から記憶を辿るのが難しく、レシートの紛失によって言った・言わないのトラブルが起きやすい。

### 1.2 アプリのコアコンセプト

> **「LINEグループと完全に同調する、思い出写真とメモが残せる共同サイフ」**

LINEグループチャットから起動するだけで自動的にメンバーが同期され、複数回の旅行プロジェクトに対応。メンバーごとの受益率を加味した自動精算と、次回の旅行への余剰金引き継ぎを自動化します。

さらに、旅行の総支出や個人の実質消費額を美しくビジュアル化し、「詳細な支払いメモ」や「レシート・思い出の写真」を支出データに紐付けてアルバムのように保存できる、楽しさと透明性を最大化したサマリー機能を提供します。

## 2. ライフサイクル & LINE連携仕様

### 2.1 グループ参加フロー（招待コード方式）

LIFFアプリは個人のLINEアカウントと連携して起動します。グループ管理には、アプリが生成する **6桁英数字の招待コード** を用いた方式を採用しています。

> **設計変更の背景**: LIFF SDK の `liff.getContext().groupId` はグループチャット内のリンクから起動した場合のみ取得できますが、LINEミニアプリ（miniapp.line.me）経由での起動では取得できないケースがあります。招待コード方式は起動経路に依存せず、確実に複数メンバーを同一グループに紐付けられます。

#### グループ参加フロー

1. **グループ作成（最初の1人）**  
   最初のメンバーがアプリを開き「グループを作成する」を選択。`POST /api/groups` が呼ばれ、グループ・管理者ユーザー・最初の旅行プロジェクトが一括作成されます。作成者は `admin` ロールとなり、6桁の招待コードが表示されます。

2. **メンバーの参加**  
   招待コードを受け取った他のメンバーがアプリを開き「招待コードで参加する」から6桁コードを入力。`POST /api/groups/join` で既存グループに `member` として追加されます。既に参加済みのユーザーは重複登録されません。

3. **LINE Botとのグループリンク**  
   LINE公式アカウント（Bot）をLINEグループチャットに追加し、チャットに招待コードを送信すると、Webhook経由でグループチャットと Tabi-Pay グループが紐付けられます（`line_group_id` の保存）。以降の積立承認通知・日次まとめはこのLINEグループに配信されます。

**LIFF起動時のグループ選択ロジック（実装）**

```typescript
// LiffProvider.tsx — 初期化後にユーザーが参加中のグループを検索
const res = await fetch(`/api/users/me?userId=${profile.userId}`);
const { groups } = await res.json();

if (groups.length === 0) {
  // 未参加 → グループ作成 or 招待コード入力画面へ
  setScreen("join");
} else if (groups.length === 1) {
  // 1グループのみ → 自動選択
  setActiveGroup(groups[0]);
} else {
  // 複数グループ → 選択画面へ
  setScreen("groupSelect");
}
```

### 2.1.1 LINE Webhook（Bot）連携仕様

LINE Bot をグループチャットに招待することで、アプリとLINEグループを双方向連携させます。

| イベント | 処理内容 |
|---|---|
| `join`（Bot参加） | Botがグループに入ったことを検知し、招待コードの送信を促すメッセージを返信 |
| `message`（テキスト） | 6桁英数字のパターンに一致するテキストを招待コードとして解釈し、`groups.line_group_id` に保存。リンク完了メッセージを返信 |

```
[LINEグループ] ─ Bot招待 ──→ join イベント ──→ 「招待コードを送信して」返信
[LINEグループ] ─ 「ABC123」 ──→ message イベント ──→ groups テーブル更新 + 完了返信
```

Webhook URL: `POST /api/line/webhook`  
署名検証: `X-Line-Signature` ヘッダーを `LINE_CHANNEL_SECRET` で HMAC-SHA256 検証（開発中はスキップ可）

### 2.2 マルチプロジェクト（旅行の遷移）と繰越処理

本アプリは、単発の旅行だけでなく、同じ3人組で「何度も何度も繰り返し利用する」ことを前提として設計されています。

```
[旅行プロジェクト A (アクティブ)] ──(管理者が終了宣言)── [精算ルートの提示 & 清算実施]
                                                                   │
                                                                   ▼
[旅行プロジェクト B (自動新規作成)] ─────(プール残高 B_pool を繰越)┘
```

1. **旅行の終了宣言**  
   旅行が終了し、すべての支払い入力が完了した段階で、管理者がダッシュボードから「旅行を完了（精算完了）する」を実行します。

2. **精算実行**  
   後述する「精算アルゴリズム」に基づき、現金立て替え分の過不足を解消するための最小送金ルートが提示され、メンバー間で精算を行います。

3. **新規プロジェクトへの遷移 & 繰越金の発生**
   - 現在のプール口座残金 $B_{pool}$ は、引き出さずにそのまま次回の旅行の「繰越初期値 $C_{init}$」として引き継がれます。
   - 新しい旅行プロジェクト $T_{next}$ がデータベース上に自動生成され、ステータスが `active` になります。

## 3. 金融計算・自動精算ロジック (Computational Engine)

お金の不公平感や計算ミスを極限までゼロにするため、不均等な割り勘や現金立て替え、前回の繰越金を統合した厳密な金融方程式を定義します。

### 3.1 変数の定義

ある1つの旅行プロジェクトにおいて、以下の通り変数を定義します。

| 変数 | 説明 |
|---|---|
| $M = \{1, 2, 3\}$ | 旅行参加メンバーの集合（3人組） |
| $C_{init}$ | 前回の旅行から引き継がれた初期プール繰越金 |
| $P_i$ | メンバー $i$ が当期に口座へ振り込み、管理者に承認された積立金の合計額 |
| $E_k$（$k=1\dots K$） | 登録された個別の支出データ |
| $A_k$ | 支出 $k$ の総額 |
| $pay_k \in M$ | 支出 $k$ を実際に決済（支払）したメンバー |
| $type_k \in \{\text{Card}, \text{Cash}\}$ | 支払方法（Card = 共通カードプール金、Cash = 個人現金立て替え） |
| $B_k \subseteq M$ | 支出 $k$ の恩恵を受けたメンバー（受益者）の集合（例：AとBの2人なら $B_k = \{A, B\}$） |

### 3.2 物理プール残高 ($B_{pool}$) の計算

共通の銀行口座・サイフの中に「物理的に残っている残高」です。

前回の繰越金と、今回全員が実際に振り込んで承認された積立金の合計から、共通カード（プール金）を使って決済した支払金額のみを差し引きます。  
（※個人現金での立て替えは共通口座から引かれていないため、ここでは減算しません）

$$B_{pool} = C_{init} + \sum_{i \in M} P_i - \sum_{type_k = \text{Card}} A_k$$

### 3.3 各自の純過不足金（Net Position $Net_i$）の計算

旅行終了時（プロジェクト完了時）に、各メンバー $i$ がグループ全体に対してどれだけ「払いすぎている（$Net_i > 0$ = 債権）」か、「もらいすぎている（$Net_i < 0$ = 債務）」かを計算します。

#### ① 各自の実際の支出貢献度 ($Contrib_i$)

各自がこの旅行のために身銭を切って貢献した総額です。「承認済の積立額 $P_i$」に「個人現金で立て替えた総額 $OutPocket_i$」を加算したものです。

$$OutPocket_i = \sum_{\substack{type_k = \text{Cash} \\ pay_k = i}} A_k$$

$$Contrib_i = P_i + OutPocket_i + \frac{C_{init}}{|M|}$$

#### ② 各自の総受益額 ($Benefit_i$)

旅行全体において、各自が消費したサービスの総価値（自分が得した額）です。一律3等分の支出だけでなく、特定のメンバーのみを対象とした不均等な支出（例：AとBだけで食べたおやつ代など）も、受益者の数 $|B_k|$ に応じて公平に等分されます。

$$Benefit_i = \sum_{\substack{k \in \text{All} \\ i \in B_k}} \frac{A_k}{|B_k|}$$

#### ③ 各自の最終プール取り分 (Asset Share)

現在の旅行が終わり、口座に残ったプール残高 $B_{pool}$ は、自動的に次回の旅行プロジェクトに等分（$\frac{B_{pool}}{|M|}$ ずつ）で引き継がれる資産となります。そのため、現時点の精算においては、各自がその分の権利を確保しているものとして取り扱います。

#### ④ 純ポジション ($Net_i$) の算出式

各自が貢献した総額から、消費した額（受益額）と、次の旅行へ引き継ぐプール取り分の資産権利を差し引いたものが、最終的な個人間の過不足金となります。

$$Net_i = Contrib_i - Benefit_i - \frac{B_{pool}}{|M|}$$

> **保存則の証明**: 全員の $Net_i$ の合計は、いかなる割り勘条件であっても、常に完璧に $0$ に収束します。

$$\sum_{i \in M} Net_i = \sum_{i \in M} Contrib_i - \sum_{i \in M} Benefit_i - B_{pool}$$

それぞれの総和を展開すると：

$$\sum_{i \in M} Contrib_i = \sum_{i \in M} P_i + \sum_{type_k=\text{Cash}} A_k + C_{init}$$

$$\sum_{i \in M} Benefit_i = \sum_{type_k=\text{Card}} A_k + \sum_{type_k=\text{Cash}} A_k$$

これらを元の式に代入します：

$$\sum_{i \in M} Net_i = \left( \sum_{i \in M} P_i + \sum_{type_k=\text{Cash}} A_k + C_{init} \right) - \left( \sum_{type_k=\text{Card}} A_k + \sum_{type_k=\text{Cash}} A_k \right) - B_{pool}$$

$$= \left( C_{init} + \sum_{i \in M} P_i - \sum_{type_k=\text{Card}} A_k \right) - B_{pool} = B_{pool} - B_{pool} = 0$$

これにより、計算上の過不足が完全に相殺され、誰一人損得なしに精算できることが数学的に証明されます。

### 3.4 精算ルーティング（債務最小化アルゴリズム）

旅行終了時の個人間送金において、送金の手間とトランザクション（振込手数料や回数）を最小にするため、以下のGreedy（貪欲）アルゴリズムを用いて最適な送金ルートを導出します。

```
[各自の過不足金 Net_i を算出]
       │
       ├─ 債権者リスト（Net_i > 0、降順） 例: [C +1000]
       └─ 債務者リスト（Net_j < 0、昇順） 例: [A -500, B -500]
       │
[最大債務者 D と 最大債権者 C をマッチング]
       │  ➔ 送金額 X = min(|Net_D|, Net_C)
       │  ➔ 「D から C へ X 円送金」のルートを生成
       │  ➔ Net の残高を更新
       ▼
[全員の Net が 0 になるまでループ] ── [完了！最適な送金ルートを出力]
```

#### アルゴリズムの実行ステップ

1. **メンバーを分類**
   - $Net_i > 0$ のメンバーを「債権者（Creditors）」リストに入れ、値が大きい順（降順）にソート。
   - $Net_j < 0$ のメンバーを「債務者（Debtors）」リストに入れ、絶対値が大きい順（昇順）にソート。

2. **マッチングのループ**
   - リスト内の最大の債務者 $D$ と、最大の債権者 $C$ を取り出します。
   - 送金する金額 $X$ を決定します：$X = \min(|Net_D|, Net_C)$
   - 送金アクションを記録：「$D$ は $C$ に $X$ 円を支払う」
   - それぞれの残高を更新します：
     - $Net_D \leftarrow Net_D + X$
     - $Net_C \leftarrow Net_C - X$
   - $Net$ の値が $0$ になったメンバーをそれぞれのリストから除外します。

3. **終了**  
   すべてのメンバーの $Net$ が $0$ になれば、生成された送金アクションのリストを精算画面に提示します（最大でも $|M| - 1 = 2$ 回の送金で完了します）。

### 3.5 旅行総支出と個人実質消費額の可視化ロジック

旅行終了時、または旅行中のサマリー画面において、「どれだけお金を使い、どれだけ楽しんだか」を盛り上げるための集計ロジックを定義します。

#### ① 旅行総支出（$TotalExpenses$）

共通カード決済分と、個人現金での立て替え分のすべてを合算した、旅行全体の「総ワクワク投資額」です。

$$TotalExpenses = \sum_{type_k = \text{Card}} A_k + \sum_{type_k = \text{Cash}} A_k$$

> **意味合い**: 「この旅行で3人でこれだけのお金を動かして楽しんだ！」という実績値を明確にし、達成感を演出します。

#### ② 決済種別ごとの合計支出

$$TotalCard = \sum_{type_k = \text{Card}} A_k \qquad TotalCash = \sum_{type_k = \text{Cash}} A_k$$

#### ③ 個人実質消費額（$Benefit_i$）

「3.3 ②」で定義した $Benefit_i$ は、割り勘処理後の「各自が実質的にいくら分旅行を楽しんだか（消費したか）」を示します。これを「個人実質消費額」として、個別のサマリーとして可視化します。

> **意味合い**: 単なる一律割とは異なり、「個別にチェックを外して精算した個別のお買い物・飲食代」が正しく加味されているため、「自分は今期の旅行で実質これだけ遊んだんだ！」というパーソナルな納得感を生み出します。

## 4. データベース設計 (Database Schema)

マルチプロジェクト（複数回の旅行）と、支出ごとの受益者選択（多対多関係）を効率的かつ整合性を保って実現するための、リレーショナルデータベース（RDB）向けテーブル設計です。

```
┌───────────┐        1:N        ┌───────────┐
│  Groups   │──────────────────│   Users   │
└───────────┘                   └───────────┘
      │                               │ 1
      │ 1:N                           │
      ▼                               │
┌───────────┐        1:N              │
│   Trips   │─────────────┐          │
└───────────┘             │          │
      │                   │          │
      │ 1:N               │ 1:N     │ N
      ▼                   ▼          ▼
┌───────────┐       ┌───────────┐    │
│  Savings  │       │ Expenses  │────┼────┐
└───────────┘       └───────────┘    │    │
                          │          │    │
                          │ 1:N      │    │
                          ▼          ▼    │
                    ┌────────────────────┐│
                    │ExpenseBeneficiaries├┘
                    └────────────────────┘
```

### 4.1 テーブル定義

#### ① Groups（旅行グループ）

Tabi-Pay のグループ単位で一意に管理される、最上位の組織テーブルです。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| `group_id` | String (UUID) | PK | アプリが生成するグループ一意ID |
| `invite_code` | String | UNIQUE | メンバー招待用6桁英数字コード（例：`ABC123`） |
| `created_by` | String | FK → users | グループ作成者の user_id |
| `approver_id` | String | FK → users / Nullable | 積立入金の承認権限者の user_id（未設定時は admin） |
| `line_group_id` | String | Nullable | LINE Bot連携で取得したLINEグループチャットのID |
| `created_at` | DateTime | — | グループの作成日時 |

#### ② Users（ユーザー）

LINEのアカウント情報を保持し、特定のグループに所属させます。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| `user_id` | String | PK | LINEから取得する userId |
| `display_name` | String | — | LINEのプロファイル名 |
| `picture_url` | String | Nullable | LINEのプロフィール画像URL |
| `group_id` | String | FK → groups | 所属するグループの group_id |
| `role` | String | — | グループ内権限（`admin`: 管理者 / `member`: 一般メンバー） |
| `created_at` | DateTime | — | ユーザーの登録日時 |

#### ③ Trips（旅行プロジェクト）

同じグループ内で、過去から現在、未来へと繰り返される個々の旅行データを管理します。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| `trip_id` | UUID | PK | 旅行プロジェクトの一意なID |
| `group_id` | String | FK | 紐づくLINEの group_id |
| `title` | String | — | 旅行の名称（例：「2026年GW沖縄旅行」） |
| `status` | String | — | プロジェクト状態（`active`: 現在進行中 / `settled`: 清算・完了済） |
| `carry_over_in` | Integer | — | 前の旅行から引き継いだ繰越金額（$C_{init}$） |
| `created_at` | DateTime | — | 旅行の作成日時 |

#### ④ Savings（積立・口座入金履歴）

メンバーが行った積立口座への入金状況を管理します。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| `saving_id` | UUID | PK | 積立の一意なID |
| `trip_id` | UUID | FK | 紐づく旅行の trip_id |
| `user_id` | String | FK | 振り込みを行ったメンバーの user_id |
| `amount` | Integer | — | 積立申請額（円） |
| `status` | String | — | 状態（`pending`: 管理者の承認待ち / `approved`: 承認・口座着金済） |
| `created_at` | DateTime | — | 振込ボタンを押した日時 |
| `approved_at` | DateTime | Nullable | 管理者が着金を確認し承認した日時 |

#### ⑤ Expenses（旅行中の支出）

旅行中に行われたあらゆる支払いのマスタデータです。タイトル、金額、決済方法に加えて、詳細な内容を書き残すメモ欄、およびレシートや思い出写真を保存する画像URLカラムを追加しました。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| `expense_id` | UUID | PK | 支出の一意なID |
| `trip_id` | UUID | FK | 紐づく旅行の trip_id |
| `payer_id` | String | FK | 実際に支払ったメンバーの user_id |
| `amount` | Integer | — | 支払総額（円）（$A_k$） |
| `payment_type` | String | — | 決済方法（`card`: 共通口座カード / `cash`: 個人現金立て替え） |
| `title` | String | — | 支出の用途（例：「レンタカー」「居酒屋代」） |
| `memo` | Text | Nullable | 支払いの詳細、割り勘の経緯、思い出メモなど |
| `image_url` | String | Nullable | アップロードされた写真・レシートの画像URL（Cloud Storage等） |
| `paid_at` | Date | — | 支払日 |
| `created_at` | DateTime | — | アプリへの入力日時 |

#### ⑥ ExpenseBeneficiaries（支出受益者 - 中間テーブル）

支出に対する割り勘対象メンバーを保持する多対多（M:N）の交差テーブルです。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| `expense_id` | UUID | PK, FK | 紐づく支出の expense_id |
| `user_id` | String | PK, FK | その支出の恩恵を受けた（割り勘対象の）メンバーの user_id |

## 5. LINE通知戦略・配信バッチ設計

LINE公式アカウントの無料プラン（月200通枠）を賢くやりくりし、重要なアクションのリアルタイム性を担保しつつ、日常の支出報告は低頻度でまとめるハイブリッド通知設計です。

### 5.1 通知トリガーとポリシー

| 通知タイプ | トリガーアクション | 配信方式 | 目的と特徴 |
|---|---|---|---|
| 即時通知（超重要） | 管理者がメンバーの積立入金を承認した瞬間 | Push Message API | 「確かに入金が反映された」という安心感の即時提供。 |
| 即時通知（超重要） | 旅行の終了・精算が確定した瞬間 | Push Message API | 「誰にいくら送金すればいいか」のルートを全員へ即座に伝達。 |
| 日次まとめ通知 | 毎晩22:00に自動実行されるバッチ処理 | Cronによる一括Push | 日中の支出登録ごとの通知をまとめ、通知疲れの防止と通数節約。 |

### 6.2 各画面のUIコンポーネント詳細

#### ① ホーム画面（ダッシュボード）

**ヘッダーエリア**
- 旅行のタイトル「2026 沖縄旅」を大きく表示。
- 「過去の旅行履歴」へアクセスできるアイコンボタン。

**ワクワク総支出（TotalExpenses）カード**
- 「現在の総支出：¥248,500」を大きく華やかに（王冠マークやクラッカーなどの絵文字付きで）表示。
- 内訳として決済種別の内訳（💳カード支払い / 💴個人立て替え）を表示。

**残高表示モジュール**
- 物理口座残高 $B_{pool}$（例：¥153,000）を表示。
- 内訳として「前回の繰越分：¥12,000」「今期の積立総額：¥150,000」「カード支払済総額：-¥9,000」を確認可能。

**メンバー積立ステータスカード**
- 3人のアイコン、名前、および「当期の積立達成状況」を色のついたバッジで表示。

**直近の支出リスト（拡張）**
- 最近登録された支出データを表示。
- 各支出行をタップすると「詳細アコーディオン」が下にスライド展開。
- 展開すると、登録された詳細メモ（テキスト）と、アップロードされたレシートまたは写真のサムネイル画像が美しく表示される。
- サムネイルをタップすると、スマートフォン全画面で拡大表示（モーダル）されるため、購入品のチェックやレシートの確認が手軽に行えます。

**画面下部固定ボタン**
- 緑色の大きな「支出を記録する（＋）」ボタン。

#### ② 支出登録画面（詳細・メモ・写真対応）

**金額・タイトル入力欄**  
「いくら使ったか（金額）」と「何に使ったか（タイトル、例：レンタカー、ステーキ、お土産）」をスムーズに入力できる1枚完結のフォーム。

**詳細メモテキストエリア（新設）**  
150文字以内の自由記入欄。「国際通りの〇〇という店」「お土産だけどC君がB君にお願いされて買ったもの」など、詳細な背景を記録可能。

**支払方法のトグルスイッチ / 支払者・受益者の選択**  
誰がカード/現金で払い、誰が割り勘の対象（受益者）かをタップで設定。

**写真・レシート登録モジュール（新設）**
- 「📷 写真を追加する」枠線のついた直感的なアップロードエリア。
- タップすると、スマホの「カメラを起動して撮影」または「ライブラリ（写真フォルダ）から選択」のオプションが起動。
- 写真が選択されると、プレビューが即座に表示され、アップロード中のプログレスバーがインジケータとして走ります。
- 不要な場合は右上にある小さな [×] ボタンで簡単に削除可能。

**保存ボタン**  
タップすると、データと画像ファイルがバックエンドに即座に送信・保存されます。

#### ③ 支出一覧画面（ExpensesScreen）

当旅行の全支出をリスト表示します。各行には金額・タイトル・日付・支払者・受益者を表示し、タップで詳細メモ・レシート写真がアコーディオン展開します。

#### ④ 旅行の締め・最終精算画面（SettleScreen）

**旅行楽しんだ度サマリー（総贅沢額表示）**  
「今回の旅行の総支出：¥248,500 💸」を表示。

**個人実質消費額（Benefit）グラフ**  
各自がこの旅行で実際に使った（満喫した）額 $Benefit_i$ を横棒グラフで表示。

**送金ナビゲーションカード**  
計算アルゴリズムによって導き出された「最も無駄のない送金アクション」を提示。

**HTMLレポートリンク（`GET /api/report?tripId=xxx`）**  
旅行の全支出・積立状況・送金ルートをまとめた印刷対応HTMLレポートを生成。ブラウザで直接開いてPDF保存ができます。

**「精算完了・プロジェクト終了」ボタン**  
現在のプロジェクトを `settled` にし、余剰金（繰越金）を引き継いだ次の旅行プロジェクトを自動立ち上げ。

#### ⑤ 旅行履歴画面（HistoryScreen）

`settled` ステータスの過去旅行一覧を表示。タップでその旅行のHTMLレポートを開けます。

#### ⑥ グループ管理（設定）

- **承認者設定（`PATCH /api/groups/:groupId/approver`）**: 管理者が積立承認権限者を任意のメンバーに変更できます。
- **グループ削除（`DELETE /api/groups/:groupId`）**: 管理者が全データ（旅行・支出・積立・ユーザー）をカスケード削除できます。
